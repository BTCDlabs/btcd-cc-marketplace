# Proposal Templates

Templates for each proposal type generated by `/optimize:discover`. All proposals follow the conventions of the [skill-creator](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/skill-creator) and [hookify](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/hookify) plugins.

## New Skill

Follow the skill-creator anatomy: SKILL.md with YAML frontmatter, optional scripts/, references/, and assets/ subdirectories.

### SKILL.md Template

```yaml
---
name: <kebab-case-name>
description: >
  <When to trigger — be specific and slightly "pushy" so Claude actually uses the skill.
  Include concrete trigger phrases like "Use when the user asks to...", "Trigger when
  code imports X", or "Use this skill whenever the user mentions Y, even if they don't
  explicitly ask for Z.">
  Do NOT trigger on <specific exclusions to prevent false positives>.
---
```

```markdown
# <Skill Name>

<Brief overview — what this skill does and why it exists>

## Workflow

<Instructions in imperative form. Explain the WHY behind each step, not rigid MUSTs.
The model is smart — give it context and reasoning so it can adapt to edge cases.>

### Step 1: <Action>
<What to do and why this matters>

### Step 2: <Action>
<What to do and why this matters>

## Output Format
<Expected output structure>

## Rules
1. <Key constraint with reasoning>
2. <Key constraint with reasoning>
```

### Key Principles (from skill-creator)

- **Description is the trigger**: Put all "when to use" info in the description, not the body. Claude decides whether to consult a skill based on the description alone.
- **Be pushy**: Claude tends to "undertrigger" skills. Make descriptions slightly aggressive about when to activate.
- **Progressive disclosure**: Keep SKILL.md body under 500 lines. Put detailed reference data in `references/` subdirectory.
- **Explain the why**: Instead of "ALWAYS use X", write "Use X because it prevents Y, which matters when Z."
- **Description under 150 words**: Respects context efficiency while being specific enough to trigger reliably.

### Directory Structure

```
<skill-name>/
├── SKILL.md          (required — frontmatter + instructions)
├── scripts/          (optional — executable code for deterministic tasks)
├── references/       (optional — detailed docs loaded on demand)
└── assets/           (optional — templates, icons, fonts)
```

## New Hook

### Hookify Rule (if hookify plugin is installed)

Create in the project's `.claude/` directory as `.claude/hookify.<rule-name>.local.md`:

```markdown
---
name: <action-verb>-<target>
enabled: true
event: <bash|file|stop|prompt|all>
pattern: <python-regex-pattern>
action: <warn|block>
---

<emoji> **<What was detected>**

<Why this matters — explain the concern clearly>

**Alternatives:**
- <Safer approach 1>
- <Safer approach 2>
```

### Advanced Hookify Rule (multiple conditions)

```markdown
---
name: <rule-name>
enabled: true
event: file
conditions:
  - field: file_path
    operator: regex_match
    pattern: <path-pattern>
  - field: new_text
    operator: contains
    pattern: <content-pattern>
action: warn
---

<Message body>
```

### Native Claude Code Hook (if hookify NOT installed)

Create a shell script at `.claude/hooks/<name>.sh`:

```bash
#!/bin/bash
set -euo pipefail

# Read hook input from stdin
HOOK_INPUT=$(cat)

# Extract relevant field
TOOL_NAME=$(echo "$HOOK_INPUT" | jq -r '.tool_name // ""')
# For Bash hooks: COMMAND=$(echo "$HOOK_INPUT" | jq -r '.tool_input.command // ""')
# For Edit hooks: FILE_PATH=$(echo "$HOOK_INPUT" | jq -r '.tool_input.file_path // ""')

# Check pattern
if echo "$FIELD" | grep -qE '<pattern>'; then
  # Block with reason
  jq -n --arg reason "<message>" '{"decision": "block", "reason": $reason}'
  exit 0
fi

# Allow by default
exit 0
```

Then add to `.claude/settings.local.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "<ToolName>",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/<name>.sh",
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

### Naming Conventions

- Start with action verb: `warn-`, `block-`, `prevent-`, `require-`, `check-`
- Be descriptive: `block-dangerous-rm`, `warn-console-log`, `require-tests-before-stop`
- Use kebab-case

## New Agent

```yaml
---
name: <agent-name>
description: >
  Use this agent for <specific task> when <trigger context>.
  <What it analyzes/produces>. Returns <output format>.
  Do NOT use for <exclusions>.
tools:
  - <minimum necessary tools>
---
```

```markdown
# <Agent Name>

<What this agent does and when it's invoked>

## Input
<What data the agent receives>

## Analysis / Workflow
<Steps the agent performs>

## Output Format
<Structured output the agent returns>
```

## Install MCP Server

Present to user (never auto-execute):

```markdown
### Install: <server-name>

**Install command:**
\`\`\`bash
<npm install -g package | npx package | pip install package>
\`\`\`

**Add to `.mcp.json`:**
\`\`\`json
{
  "mcpServers": {
    "<server-name>": {
      "command": "<command>",
      "args": [<args>],
      "env": {
        "<KEY>": "<value or instructions>"
      }
    }
  }
}
\`\`\`

**Security notes:**
- <Transport: stdio/HTTPS>
- <Credentials: API key required? How to obtain?>
- <Access scope: what can this server read/write?>
- <Recommendation: deferred loading if many tools>
```

## Remove Item

```markdown
### Remove: <item-type> — <item-name>

**What gets deleted:**
- <file path(s)>

**Evidence for removal:**
- <Zero usage across N sessions>
- <No references found in config files>

**Safety check:**
- [ ] Not security-related
- [ ] No dependencies found
- [ ] Older than 30 days
- [ ] User explicitly approved

**Reversibility:**
- <How to restore if needed — e.g., reinstall command, git checkout>
```
