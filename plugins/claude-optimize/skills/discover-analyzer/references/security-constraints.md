# Security Constraints for Discovery Proposals

Every proposal generated by `/optimize:discover` must satisfy these constraints. The discovery-proposer agent checks each proposal against these rules and assigns a security assessment: PASS ✓, WARN ⚠, or BLOCK ✗.

**IMPORTANT**: Only BLOCK ✗ proposals are filtered out. WARN ⚠ proposals MUST still be presented to the user with the warning details — the user decides whether to proceed. Do NOT self-censor WARN proposals. An MCP server requiring an API key is WARN, not BLOCK. A capability that overlaps partially with an existing tool is still worth proposing if it adds value.

## New Skill Constraints

### Forbidden

- `Bash(*)` in allowed-tools — must scope to specific commands (e.g., `Bash(npm:*)`, `Bash(python3:*)`)
- Reading or writing credential files: `.env`, `.key`, `.pem`, `.p12`, `credentials.json`, `secrets.*`, `*.secret`
- Instructions that bypass safety hooks or deny rules (e.g., "ignore the block-sensitive-writes hook")
- `disable-model-invocation` without clear justification and user awareness
- Overly broad file access patterns that could touch sensitive directories

### Required

- "Do NOT trigger on" clause in description — prevents false positive activation
- Description under 150 words — respects context efficiency
- Allowed-tools scoped to the minimum set necessary for the skill's function
- If the skill uses Bash, each command pattern must be explicitly listed

### Assessment Rules

| Condition | Assessment |
|-----------|------------|
| All constraints met | PASS ✓ |
| Broad but justifiable Bash patterns (e.g., `Bash(git:*)`) | WARN ⚠ — note the breadth |
| Any forbidden pattern present | BLOCK ✗ — do not present |

## New Hook Constraints

### Required for Shell Scripts

- `set -euo pipefail` at the top of every script
- All variables must be quoted (`"$VAR"` not `$VAR`)
- Input validation for hook JSON (check with `jq` before processing)
- Exit code 0 for allow, non-zero only when intentionally blocking

### Required for Hookify Rules

- `enabled: true` in frontmatter (user can disable later)
- Clear, specific regex patterns — avoid overly broad matches like `.*`
- Message body explains what was detected AND why it matters
- Suggests alternatives, not just blocks

### Forbidden

- Scripts that modify files (hooks should observe, not mutate)
- Bypassing or disabling existing security hooks
- Patterns so broad they would trigger on every operation

## MCP Server Constraints

### Trust Verification

Before proposing any MCP server installation:

1. **Source**: Must be open source (GitHub/GitLab) OR from a known publisher (Anthropic, major cloud providers, established companies)
2. **Transport**: HTTPS required for remote servers. Local stdio servers are acceptable.
3. **Credentials**: Flag all API key or token requirements as WARN ⚠
4. **Permissions**: Assess what the server can access — filesystem, network, databases

### Assessment Rules

| Condition | Assessment |
|-----------|------------|
| Open source, no credentials, local transport | PASS ✓ |
| Requires API key but from trusted publisher | WARN ⚠ — flag key requirement |
| Hardcoded secrets in configuration | BLOCK ✗ |
| Unknown publisher, closed source | BLOCK ✗ |
| Requires broad filesystem access | WARN ⚠ — note access scope |

## Removal Constraints

### Never Propose Removal Of

These categories are **exempt from bloat detection** regardless of usage data:

- Hooks with "security", "block", "deny", "sensitive", "dangerous", "protect" in name or description
- Skills related to security auditing (security-auditor, security-scanner, etc.)
- Deny rules in `.claude/settings.json` or `.claude/settings.local.json`
- MCP servers that provide security functionality (vulnerability scanners, secret detectors)
- Any item that is a dependency of a security-related item

### Safe Removal Criteria

Before proposing removal of non-security items:

1. **Evidence threshold**: Zero usage across 15+ analyzed sessions
2. **Dependency check**: Grep all config files, command prompts, skill descriptions, and agent prompts for references
3. **Recency check**: Item must be older than 30 days (recently added items get a pass)
4. **User-created vs plugin**: Plugin-provided items need stronger evidence (they were deliberately installed)

## Config Change Constraints

- Never propose weakening deny rules (removing entries or broadening patterns)
- Never propose adding `Bash(*)` to allow rules
- Permission broadening proposals are always WARN ⚠ with explicit justification
- Permission tightening proposals are always PASS ✓
